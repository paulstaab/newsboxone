# Frontend Integration Test Cases Baseline

## Purpose

This document catalogs the frontend integration tests that should exist for the NewsBoxOne frontend.
It is organized by shared shell behavior and page so coverage can scale as more pages are added.

Frontend unit tests should still be created whenever practical, but they are intentionally not tracked in this top-level document.

## Scope

This document is the maintained baseline for frontend integration tests only.

Frontend integration tests in this document should:

- run the frontend standalone
- mock the backend API instead of depending on a live backend
- focus on browser-visible frontend behavior, client-state transitions, accessibility, and browser-platform interactions that do not require a live backend journey

Prefer live-backend end-to-end scenarios in `docs/e2e-test-cases.md` over frontend integration tests whenever a cross-stack journey can reasonably be tested through the combined product.
Broad authentication, feed-management CRUD, and persisted reading-flow journeys should therefore be tracked in `docs/e2e-test-cases.md`, while this document stays focused on standalone frontend behavior with mocked API responses.

## Source of Truth

- `frontend/tests/integration/`
- `frontend/tests/visual/`

## Test Case IDs

- `TC-APP-*`: shared shell and cross-page behavior
- `TC-LOGIN-*`: login page
- `TC-FEEDS-*`: feed management page
- `TC-TIMELINE-*`: timeline page

## Current Gaps

- The current automated suite does not yet provide strong frontend integration coverage for queue docking or focus trap edge cases.
- Frontend integration coverage is intentionally narrow on feed-management mutations because live-backend CRUD journeys are tracked in `docs/e2e-test-cases.md`.
- Frontend unit tests are expected where practical, but their inventory is intentionally not tracked in this document.
