# Backend Integration Test Cases Baseline

## Purpose

This document catalogs the backend integration tests that should exist for NewsBoxOne.
Each test case has a stable ID, a short description, and an expected result so it can be understood without opening source files.

Backend unit tests should still be created whenever practical, but they are intentionally not tracked in this top-level document.

## Scope

This document is the maintained baseline for backend integration tests only.

It should cover behavior that exercises multiple backend components together, such as:

- public API routes
- persistence and migrations
- refresh and ingestion pipelines
- CLI-driven backend workflows
- service-runtime behavior

## Execution Model

The public API is specified in `docs/api-contract.yaml`.
Integration tests in this document should be based on the backend behavior tracked here.

Rust backend integration tests should seed fixture data on top of the production SQL migrations so schema coverage stays aligned with runtime behavior.

## Exclusions

- Backend unit tests belong in the backend test suite, but are not cataloged here.
- Cross-stack browser journeys belong in `docs/e2e-test-cases.md`, not in this document.
- Helper-level behaviors such as HTML-cleanup edge cases, thumbnail-extraction edge cases, prompt building, and parser normalization should normally be covered by backend unit tests unless they are exercised through a broader integration flow.

## Source of Truth

- Backend integration tests under `tests/`.
- Shared fixtures under `tests/fixtures/`.
- Shared API behavior in `docs/api-contract.yaml`.
