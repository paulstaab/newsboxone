# Implemented Test Scenarios Baseline

## Purpose

This document summarizes the main end-to-end user journeys covered by the current test suite.
It is grouped by page so more pages can be added later.

## Shared Preconditions

- The frontend is running with Playwright API mocks unless the scenario says otherwise.
- Authentication state starts either empty or from `tests/e2e/.auth/user.json`, depending on the journey.
- Backend interactions target the NewsBoxOne API endpoints expected by the frontend.

## End-to-End Scenarios

### Shared App Shell

| ID           | Journey                                | Preconditions                                      | Actions                                                                                                      | Expected Result                                                                                         |
| ------------ | -------------------------------------- | -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------- |
| `TS-APP-001` | Root route redirects to the right page | App starts with either an empty or stored session. | Open `/`.                                                                                                    | The app redirects to `/login` when signed out and `/timeline` when signed in.                           |
| `TS-APP-002` | Shared shell remains accessible        | App is loaded on login or timeline.                | Use keyboard navigation and run axe checks.                                                                  | Skip link, focus order, labels, contrast, and baseline accessibility checks pass.                       |
| `TS-APP-003` | PWA install prompt lifecycle           | Browser provides install capability.               | Open the app, dismiss the prompt, reload within and after the cooldown window, then dispatch `appinstalled`. | Prompt appearance, cooldown suppression, settings entry, and installed-state hiding behave as expected. |
| `TS-APP-004` | Offline feedback from the shared shell | Timeline is open.                                  | Simulate browser offline and online events.                                                                  | Offline banner appears when offline and hides again after connectivity returns.                         |

### Login Page

| ID             | Journey                                                | Preconditions                                  | Actions                                                                      | Expected Result                                                                                     |
| -------------- | ------------------------------------------------------ | ---------------------------------------------- | ---------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| `TS-LOGIN-001` | First-time user completes guided login                 | Storage is empty and API mocks return success. | Open `/`, enter server URL, continue, enter credentials, and submit.         | The wizard validates the server, authenticates, persists the session, and redirects to `/timeline`. |
| `TS-LOGIN-002` | User sees connectivity error before credentials        | Storage is empty and version check fails.      | Enter an unreachable or invalid server URL and continue.                     | The page stays on the server step and shows an actionable validation error.                         |
| `TS-LOGIN-003` | User is blocked from insecure server URLs              | Storage is empty.                              | Enter an `http://` URL and continue.                                         | The login flow rejects the URL and explains the HTTPS requirement.                                  |
| `TS-LOGIN-004` | User chooses whether the session persists past the tab | Storage is empty and login succeeds.           | Complete login once with the default setting and once with remember enabled. | The default session lands in `sessionStorage`; remembered login lands in `localStorage`.            |

### Timeline Page

| ID                | Journey                                                 | Preconditions                                                                      | Actions                                                              | Expected Result                                                                                                        |
| ----------------- | ------------------------------------------------------- | ---------------------------------------------------------------------------------- | -------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `TS-TIMELINE-001` | Signed-in user lands on the highest-priority folder     | Stored auth exists and timeline mocks return unread items across multiple folders. | Open `/timeline`.                                                    | The folder with the highest unread priority is active and only its cards are shown initially.                          |
| `TS-TIMELINE-002` | Reader works through folders with mark all read         | Timeline contains multiple folders with unread cards.                              | Open `/timeline`, mark the current folder read, and continue.        | The bulk read API is called, the current folder leaves the queue, and the next folder becomes active.                  |
| `TS-TIMELINE-003` | Reader skips folders and restarts the queue             | Timeline contains multiple queued folders.                                         | Repeatedly skip folders until the queue is exhausted, then restart.  | Each skipped folder moves back, “All folders viewed” appears at the end, and restart restores the queue.               |
| `TS-TIMELINE-004` | Timeline refreshes without throwing away local progress | Timeline has cached items and API mocks can return updated or failing responses.   | Load `/timeline`, let auto-refresh run, then manually refresh again. | New data merges into the cache, refresh failures preserve the existing view, and action controls recover after errors. |
| `TS-TIMELINE-005` | Reader opens an article without losing place            | Timeline has at least one unread article.                                          | Click an article card, inspect the pop-out, close it, then refresh.  | Opening the card marks it read and shows the pop-out; the item stays visible until refresh reconciliation removes it.  |
| `TS-TIMELINE-006` | Keyboard user drives queue actions from the page        | Timeline is open and body focus is outside input controls.                         | Press `r`, `ArrowRight`, and `Enter`.                                | Refresh, skip, and mark all read execute through hotkeys and move the queue forward.                                   |
| `TS-TIMELINE-007` | Queue state survives a reload                           | A folder was already marked read in the current browser storage.                   | Reload `/timeline`.                                                  | The next folder remains active and the already-processed folder does not reappear immediately.                         |

| `TS-FEEDS-004` | Signed-in reader scans compact feed status rows             | Stored auth exists and API mocks provide existing folders and feeds.      | Open `/feeds` and inspect the grouped table, including a folder subheader row and an existing feed row with and without update failures. | The page renders a five-column feed table, merges folder subheaders across the table width, shows relative timestamps without repeated row labels, exposes exact local datetimes on timestamp hover, exposes feed URLs on hover, and shows healthy/error status icons separate from row actions. |

| `TS-FEEDS-006` | Reader deletes a folder with assigned feeds                 | Stored auth exists and the page contains a folder with at least one feed. | Delete the folder and accept the confirmation warning.                                                                                   | The page warns that assigned feeds will be unsubscribed, deletes those feeds, and removes the folder.                                                                                                                                                                                            |
